## Why this batch

Stage S1 (✅) delivered the dependency-free data core: every on-wire JSON-RPC + MCP shape as a struct + contract, the constants, the guards, the N1 normalized-superset façade, and the bidirectional `exn`↔JSON-RPC error layer. **Stage S2 completes layer L0** by building everything that sits directly on top of S1 but still below any transport or engine: the pluggable JSON-Schema validator provider with a Racket-native default, the dual-form (contract *or* JSON Schema) schema-normalization util, and the cohesive shared-utility modules (URI templates, tool-name validation, `_meta` metadata, shared auth structs, stdio framing). These are the last L0 pieces; completing S2 unblocks Stage S3 (transport port + in-memory adapter), which imports only L0.

S2's modules split cleanly into testable sub-multi-day units: the validator port (M3) and its default provider (M3) are separated because the provider's documented-keyword-subset work is substantial; the schema-normalization util (M4) is its own unit because it bridges contracts and JSON Schema and delegates to M3; the small M5 utilities (M5a–M5e) are grouped by cohesion and size — URI templates (M5a) and tool-name validation (M5b) each earn their own item against TS test fixtures, while the smaller `_meta` metadata (M5c) and shared-auth (M5d) modules are grouped, and stdio framing (M5e) is its own item with a standalone harness.

**M5e is orphaned until S6a.** `shared/stdio.rkt` (M5e — newline-delimited JSON framing) has **no consumer inside S2**: its first real consumer is the stdio transport (M7) in **S6a**. It is built here for L0 cohesion (architecture groups it under M5) and is **unit-tested standalone** — multi-message and partial-frame buffering against a synthetic byte stream — with integration coverage arriving alongside M7 in S6a. (An implementer who prefers to defer M5e may move it beside M7 in S6a without affecting any other S2 deliverable; this queue builds it now.)

**Prerequisite for the Worker:** the TS reference checkout at `typescript-sdk/` must be available, since several acceptance criteria assert behavioural parity — URI-template round-trips vs `test/shared/uriTemplate.test.ts`, tool-name accept/reject parity vs `test/shared/toolNameValidation.test.ts`, and validator results cross-checked against the keyword semantics asserted in `test/validators/validators.test.ts`. Every S2 module imports **only** S1 (types M1 + errors M2); none may pull in a transport, engine, or role module.

### Item 010: Validator-provider port (M3)
Implement `mcp/core/validators/provider.rkt` — the pluggable JSON-Schema validator-provider interface via `racket/generic` (`gen:`-style). The source interface is TS `validators/types.ts`, whose `jsonSchemaValidator` is a **single fused method** `getValidator(schema) → (input) => result` (compile+validate in one). The Racket port **deliberately splits this into two ops** — **compile** a JSON Schema → a reusable validator handle, and **validate** a value against a compiled handle → an ok / errors result (with a structured error shape carrying path + message) — an intentional, more-idiomatic factoring that lets a handle be compiled once and reused, **not** a 1:1 mirror of the fused TS method. This is the seam that lets the default Racket-native provider (item 011) and any future vetted-library provider be swapped without changing callers. Provide an explicit `provide`d public surface; the module imports only S1 (types + errors). Testable: a unit test under `mcp/core/validators/test/` defines a trivial stub provider satisfying the generic interface, compiles a schema through it, validates an ok and an error value, and asserts the result shape (ok vs structured errors); `raco test` passes.

### Item 011: Default Racket-native provider — `from-json-schema` (M3)
Implement `mcp/core/validators/from-json-schema.rkt` — the default provider implementing the item-010 port. **Framing:** TS `fromJsonSchema.ts` is a ~43-line, keyword-**free** wrapper (schema-in → validate-fn-out); the actual keyword-validation logic in TS lives in the Ajv / cfWorker providers, which vision §8 **excludes** and §4.5 collapses into one Racket-native provider. So this module mirrors `fromJsonSchema.ts` only for the **wrapper shape**; its **keyword semantics are net-new Racket-native design** (the Ajv/cfWorker collapse per §4.5/§8), targeting the behaviour that `test/validators/validators.test.ts` asserts. Record the hand-rolled-subset-vs-library decision in the module docs as a justified Minimal-deps choice (default = hand-rolled subset unless a vetted lib is adopted). **Minimum supported keyword set:** `type` (`string`/`number`/`integer`/`boolean`/`object`/`array`/`null`), `properties`, `required`, `enum`, `items`, and `format` for the common string formats (`date-time`, `uri`, `email`). **Deliberately deferred (common) keywords — documented as unsupported in this iteration:** `pattern`, `minLength`/`maxLength`, `minimum`/`maximum`, `additionalProperties`, `uniqueItems`. Each unsupported keyword is handled per a single documented policy (ignore-with-warning **or** reject) — never silently mis-validated. Testable: a suite under `mcp/core/validators/test/` includes **at least one accept + one reject case for each** of `type`, `object`/`properties`, `required`, `enum`, and `string`-`format`, each cross-checked against the keyword semantics in `test/validators/validators.test.ts` for the same schema + value; a further test asserts that **each** of the named deferred keywords (`pattern`, `minLength`/`maxLength`, `minimum`/`maximum`, `additionalProperties`, `uniqueItems`) is handled per the documented unsupported-keyword policy and is listed in the module docs; `raco test` passes.

### Item 012: Schema-normalization util — contract-or-JSON-Schema (M4)

### Item 015: `_meta` metadata utils + shared `AuthInfo` (M5c + M5d)
Implement two cohesive shared modules.

**(1) `mcp/core/shared/metadata-utils.rkt` (M5c)** — genuinely mirror TS `metadataUtils.ts`, which is a 26-line module containing **only** `getDisplayName` (display-name precedence: `title` → `annotations.title` → `name`). **Provide `get-display-name`** with that exact precedence — the high-level server (M12b / S6b) needs it anyway. Additionally expose accessors/setters for the **reserved `_meta` keys**, respecting the reserved-key namespace and round-tripping with the S1 `_meta` envelope from the `2026-07-28` spec types (item 004). **Reserved-key count reconciliation (5-vs-8 — flag, do not lose):** `types/constants.ts` defines **eight** reserved `_meta` keys — the five S1 captured (protocol version, client info, client capabilities, related-task, deprecated log level) **plus three W3C trace-context keys `traceparent` / `tracestate` / `baggage` (SEP-414)** that S1 did **not** capture. Resolution for this item: **define the three missing trace-context key constants in M5c** (or, if the implementer prefers, scope them out of S2 and file an explicit S1 follow-up) — either way the 5-vs-8 discrepancy is documented in the module so it is not silently dropped. The SDK does not interpret trace-context values; they pass through `_meta` untouched.

**(2) `mcp/core/shared/auth.rkt` (M5d)** — the **`AuthInfo` struct** + token/metadata **helpers**. Note the split: the `AuthInfo` shape itself is defined in TS at `types/types.ts:435` (not in `auth.ts`); the `shared/auth.ts` + `authUtils.ts` pointer covers the **token/metadata helpers only**. `AuthInfo` field surface to mirror: `token`, `clientId`, `scopes`, optional `expiresAt`, optional `resource` (a URL), optional `extra`. This is the shared struct the S8 client + server auth will both consume (no OAuth logic here — struct + helpers only). Both modules import only S1.

Testable: under `mcp/core/shared/test/`, (a) a `get-display-name` test asserting the `title` → `annotations.title` → `name` precedence (incl. the empty-string-title fallthrough); (b) a `_meta` test that writes each reserved key, reads it back, asserts a non-reserved key is left untouched, and asserts the three trace-context constants exist (or that the S1 follow-up is filed); (c) an `AuthInfo` test that constructs the struct, exercises the token/metadata helpers, and asserts the field surface is exactly `token`, `clientId`, `scopes`, `expiresAt` (optional), `resource` (optional URL), `extra` (optional); `raco test` passes.

### Item 016: stdio framing — newline-delimited JSON (M5e, orphaned until S6a)
Implement `mcp/core/shared/stdio.rkt` (M5e) — newline-delimited JSON **frame encode/decode over a byte stream**, mirroring TS `stdio.ts` (its `ReadBuffer`). This is the only M5 module performing I/O. Provide an encoder (message → framed bytes) and a decoder/buffer that consumes a byte stream and yields complete messages, **buffering partial frames** across reads. **Mirror these three `ReadBuffer` behaviours (do not omit, do not "fix"):** (a) **max-buffer-size enforcement** — a configurable max with default `STDIO_DEFAULT_MAX_BUFFER_SIZE = 10 MB` (`10 * 1024 * 1024`); appending past it **throws** (DoS guard); (b) **CRLF tolerance** — strip a trailing `\r` before parse so `\r\n`-framed lines work; (c) **skip non-JSON lines** — a line that fails to parse as JSON is **skipped (continue), NOT a hard error** (intentional — flagged so the Worker doesn't "correct" it to a throw). **Orphaned-until-S6a note:** this module has **no consumer in S2** — its first real consumer is the stdio transport (M7) in S6a; it is built here for L0 cohesion and **unit-tested standalone**, with integration coverage arriving with M7. Imports only S1. Testable: under `mcp/core/shared/test/` (porting fixtures from `typescript-sdk/packages/core/test/shared/stdio.test.ts`), a standalone harness asserting: a **multi-message** round-trip (encode N messages, feed the concatenated bytes, assert all N decode in order); **partial-frame buffering** (stream split mid-frame across two reads reassembles correctly); **max-buffer overflow throws** (append past 10 MB / a small configured max raises); **CRLF tolerance** (a `\r\n`-framed message decodes identically to its `\n`-framed form); and **non-JSON-line skipping** (a non-JSON line between two valid frames is skipped, both valid frames still decode); `raco test` passes.

# Execute Work Item

Implement a work item as specified in the docs/aide/items/ directory.

## Purpose

This is Step 7 of the AI-Driven Engineering workflow. This step takes a detailed work item specification and implements it — writing code, tests, configuration, and documentation as specified.

## User Input

{{args}}

## Arguments

Parse the arguments above for the `--auto` token:

- **`--auto` present** → enable **Auto Mode**. Strip the token; whatever text remains (if any) is the item number, interpreted exactly as below. In Auto Mode, once the item is implemented you automatically advance the pipeline instead of handing back to the user. See **Auto Mode** under *Next Step*.
- **`--auto` absent** → default manual mode. The full argument (if any) is the item number, and behavior is unchanged.

## Instructions

**Efficiency contract:** follow `docs/aide/EFFICIENCY.md` (R1–R5). The main thread carries the whole roadmap's context (no `/clear` in `--auto`), so it must stay lean: read ranges not whole files, and hand subagents thin, fact-carrying prompts.

### Item Selection

If `{{args}}` is provided, treat it as an item number. Find the matching file in `docs/aide/items/` (e.g., item 5 maps to `docs/aide/items/005-*.md`).

If `{{args}}` is empty, automatically pick the next item (lean reads — R1):
1. `Grep` the status markers in `docs/aide/progress.md` (the 📋/🚧/✅ rows) instead of reading the whole file; list existing files in `docs/aide/items/` with a directory listing, not by reading them
2. Select the first work item whose status is 📋 (Planned) — it has a spec but hasn't been started
3. Tell the user which item was auto-selected before proceeding

### During Implementation

1. **Follow the specification** — implement exactly what the work item describes
2. **Document decisions** — as you make implementation choices, UPDATE the work item's "Decisions & Trade-offs" section with:
   - What was decided
   - Why this approach over alternatives
   - Any trade-offs or future considerations
3. **Update progress** — update `docs/aide/progress.md` status:
   - 📋 → 🚧 when starting implementation
   - 🚧 → ✅ when implementation is complete
4. **Scope your updates** — only update progress rows that correspond to YOUR item number. Do NOT mark other items as complete, even if their criteria happen to be satisfied as a side effect of your work. Each item must go through its own create-item → execute-item cycle.

### On Smooth Completion

- No feedback loop needed
- Ensure work item decisions are documented
- Mark progress as complete

### On Issues

If you encounter problems (unclear requirements, blocked, need help):
- Document the issue in the work item, and append a one-line signal to `docs/aide/feedback/signals.md` (what blocked you, item number) so the feedback pass can act on it
- **Auto mode:** the per-item `feedback-loop --auto` pass that runs after this item will pick up the signal and triage it (escalating to deep analysis if systemic)
- **Manual mode:** tell the user to run `/speckit.aide.feedback-loop` to adjust the process

### Worker/Reviewer Team Loop

This command participates in the worker/reviewer loop. Instead of spawning a fresh one-shot subagent for each pass, create a **persistent team** of a Worker and a Reviewer so that both retain context across iterations.

1. **Team Setup (once per invocation)**: Call `TeamCreate` with `team_name: "aide-execute"`. Spawn a persistent **Worker** via the `Agent` tool (`team_name: "aide-execute"`, `name: "worker"`, `subagent_type: "general-purpose"`, `model: "sonnet"` — medium effort) whose standing role is to implement the selected work item per its specification — writing code, tests, config, and docs, documenting decisions in the item, and updating `docs/aide/progress.md` (status cell only — R5) — then `SendMessage` `team-lead` when done. Tell it to obey `docs/aide/EFFICIENCY.md` (read ranges, R1; reference code by path:line). Spawn a persistent **Reviewer** via the `Agent` tool (`team_name: "aide-execute"`, `name: "reviewer"`, `subagent_type: "code-reviewer-expert"`, `model: "opus"` — high effort) whose standing role is to review the implementation against the work item's acceptance criteria and report back.
2. Create or update `docs/aide/versions/<command-name>/iteration-NNN/` before work begins.
3. **Worker Pass** (thin handoff — R4): `SendMessage` to `worker` with the item path and the specific excerpts/feedback it needs (and prior reviewer feedback verbatim if NNN > 001). Wait for completion, then record a `pointer.txt` (canonical item path) and, if useful, a unified `git diff` of changed files — not full-file copies (R3).
4. **Reviewer Pass**: `SendMessage` to `reviewer` pointing at the canonical files/sections (R1); have it save feedback to `docs/aide/versions/<command-name>/iteration-NNN/reviewer-feedback.md` and store the reviewer result in `reviewer-feedback.json` when JSON is produced.
5. If the reviewer has actionable criticism, increment the iteration number and return to the Worker Pass — reusing the **same** teammates so they keep their context.
6. If the reviewer has nothing left to criticize, mark this iteration as approved in `docs/aide/versions/<command-name>/approved-version.md`, shut down both teammates via `SendMessage` `{type: "shutdown_request"}`, and call `TeamDelete`.
7. Preserve the original output path for this command as the latest approved artifact, while keeping every rejected/intermediate version in the version directory.

## Next Step

**Manual mode (default — no `--auto`):**

- **More items in queue?** Start a **new chat session** and run `/speckit.aide.create-item` for the next queue item, then `/speckit.aide.execute-item` to implement it.
- **Queue exhausted?** Start a **new chat session** and run `/speckit.aide.create-queue` to generate the next batch.
- **All stages complete?** The project is done!

**Auto Mode (`--auto`):** This step is the engine that keeps the pipeline turning. Once the item is implemented and `docs/aide/progress.md` is marked ✅ (and the Reviewer has approved), do not hand back to the user.

**First, run the per-item feedback pass.** Invoke the **Skill** tool: call `speckit.aide.feedback-loop` with args `--auto <item-number>`. This is a lean triage (see that skill) that may tune the AIDE skills/docs for efficiency; it does **not** advance the pipeline and returns control here (Protected invariant P2). Wait for it to finish, then continue.

Next, decide the next action. Read leanly (R1): `Grep` the status rows of `docs/aide/progress.md`, list `docs/aide/queue/` and `docs/aide/items/`, and consult `docs/aide/roadmap.md` by section — do not read these in full each hop.

1. **Roadmap fully complete?** (every milestone/item in the roadmap is done) → **STOP**. Report to the user that the project is complete. Do not chain. This is the termination condition that prevents a runaway loop (Protected invariant P1).
2. **Current queue still has items lacking a work-item spec?** → invoke the **Skill** tool: call `speckit.aide.create-item` with args `--auto`. With no item number it auto-selects the next un-specified item, then chains back here.
3. **Queue exhausted but roadmap still has incomplete milestones?** → invoke the **Skill** tool: call `speckit.aide.create-queue` with args `--auto` to generate the next batch, which then chains into create-item.

Always re-check the termination condition (step 1) before chaining — never start another step once the roadmap shows everything complete. The chain runs continuously until then or until the user interrupts; because each step delegates its heavy work to Worker/Reviewer subagents, the main thread stays lean across the loop.

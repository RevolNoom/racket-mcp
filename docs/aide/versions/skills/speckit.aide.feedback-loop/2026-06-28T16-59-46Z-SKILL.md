# Feedback Loop (self-tuning)

Analyze what went wrong **and what cost too much**, then improve the process by
editing the AIDE skills and docs directly.

## Purpose

Step 8 of the AIDE workflow, and the only step allowed to modify the other
skills. It exists because the `--auto` chain runs unattended with no `/clear`,
so process gaps and token waste compound silently. This step catches both and
fixes them at the source.

## Arguments

Parse the arguments above:

- **`--auto` present** → **Triage Mode** (lean, cheap, runs every item from the

- **`--auto` absent** → **Deep Mode** (manual, thorough). Full analysis with the
  worker/reviewer team loop.

Before doing anything that edits a skill, read `docs/aide/feedback/PROTECTED.md`.

## Triage Mode (`--auto`, per item)

Goal: spend almost nothing when things are healthy; act only on real signals.
Do **not** spawn a team in triage. Keep total work to a handful of bounded
tool calls.

### 1. Gather signals (bounded reads only)

- **Process signals:** read `docs/aide/feedback/signals.md` if it exists (issues
  appended by execute-item). Read the just-completed item's
  `docs/aide/versions/speckit.aide.create-item/iteration-*/reviewer-feedback.json`
  count — **> 3 iterations** to approve a spec is a signal the spec template or
  requirements are unclear.
- **Efficiency signals (audit EFFICIENCY.md R1–R5 with cheap checks):**
  - `wc -l` / size of `docs/aide/items/NNN-*.md` for the current item — **over
    ~400 lines / ~40 KB violates R2** (spec bloat).
  - size of `docs/aide/progress.md` — narrative creeping in violates R5.
  - any full-artifact copies (e.g. `item.md`, `*-version.md` with full content)
    under `docs/aide/versions/.../iteration-*/` violate R3.
  - repeated whole-file reads of the same large doc in the recent transcript
    violate R1/R4 (check only if a signal already points there).

### 2. Decide

- **No signal** → write nothing, return control to caller immediately. This is
  the common, cheap path.
- **Additive efficiency fix** (trim a bloated template, tighten a handoff or
  size rule in a SKILL.md or `EFFICIENCY.md`, convert a snapshot step to a
  pointer) → apply it now via **Skill-Edit Procedure** below, then return.
- **Control-flow / structural change, or a protected invariant is implicated**
  → do NOT apply. Append the proposal to `docs/aide/feedback/proposals.md` and
  return (P5). A human decides.
- **Systemic process issue** (recurring across items, needs redesign) →
  escalate: run **Deep Mode** below, then return.

After acting (or not), return control to execute-item. Never call create-item,
create-queue, or execute-item from here (P2).

## Deep Mode (manual, or escalated from triage)

Use this for substantial redesign: missing vision/roadmap structure, a command
that repeatedly produces bad output, a process that keeps needing human rescue.

### What to analyze

1. **Document gaps** — what should have been in `vision.md` / `roadmap.md` /
   `progress.md` / the item spec but wasn't?
2. **Process issues** — why did a human have to intervene? unclear requirements?
   dependencies not surfaced upfront? scope creep?
3. **Efficiency** — which of R1–R5 is being violated, where, and how much does it
   cost? Quantify (file sizes, iteration counts, re-read multipliers).
4. **Command/skill adaptations** — should a skill template change for this
   project (e.g. add an API-contract section; trim an unused one)?

### Recommendations → action

Be **minimal**: the smallest change that prevents recurrence. Then apply it via
the Skill-Edit Procedure (or proposals.md if protected). Routine per-item
decisions belong in the item's "Decisions" section, not here — this step is for
**systemic** fixes.

### Worker/Reviewer team loop (Deep Mode only)

For non-trivial redesign, use a persistent team so context is retained:

1. `TeamCreate` `team_name: "aide-feedback"`. Spawn a persistent **Worker**
   (`Agent`, `name: "worker"`, `subagent_type: "general-purpose"`) to produce
   and apply the minimal improvements, and a **Reviewer**
   (`name: "reviewer"`, `subagent_type: "general-purpose"`) to verify they are
   minimal, actionable, and safe (respect PROTECTED.md). Hand both a thin
   prompt (R4) naming the exact files and signals.
2. Iterate worker → reviewer in `docs/aide/versions/speckit.aide.feedback-loop/iteration-NNN/`,
   storing only `reviewer-feedback.md` + `.json` + a `pointer.txt` (R3).
3. When the reviewer has nothing left, write `approved-version.txt` (pointer),
   `SendMessage` `{type: "shutdown_request"}` to both, and `TeamDelete`.

## Skill-Edit Procedure (the self-modification authority)

This is the only place AIDE skills get edited. Full authority, bounded by
`docs/aide/feedback/PROTECTED.md`. For **every** edit:

1. **Check protection (P1–P5).** If the change would touch the termination guard
   (P1), the single-engine rule (P2), or is a control-flow/structural change
   (P5) → append it to `docs/aide/feedback/proposals.md` and STOP. Do not apply.
2. **Back up (P3).** Copy the target `SKILL.md` (or doc) to
   `docs/aide/versions/skills/<skill-name>/<UTC-timestamp>-SKILL.md`.
3. **Apply** the minimal edit with the `Edit` tool.
4. **Validate (P4).** The file still has a valid `---` frontmatter block with
   `name:` and `description:` intact.
5. **Log (P3).** Append an entry to `docs/aide/feedback/changelog.md`: trigger,
   change (mark additive vs control-flow), backup path, estimated token impact.

Tighten `docs/aide/EFFICIENCY.md` itself the same way when you find a new class
of waste — it is the standing contract every other skill follows.

## Next Step

- **Triage Mode (`--auto`):** return control to `execute-item`, which continues
  its termination check and chain. Do not start a new session.
- **Deep Mode (manual):** after applying changes, resume the workflow where you
  left off.

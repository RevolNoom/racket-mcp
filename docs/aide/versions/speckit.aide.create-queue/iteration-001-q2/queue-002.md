## Why this batch

Stage S1 (✅) delivered the dependency-free data core: every on-wire JSON-RPC + MCP shape as a struct + contract, the constants, the guards, the N1 normalized-superset façade, and the bidirectional `exn`↔JSON-RPC error layer. **Stage S2 completes layer L0** by building everything that sits directly on top of S1 but still below any transport or engine: the pluggable JSON-Schema validator provider with a Racket-native default, the dual-form (contract *or* JSON Schema) schema-normalization util, and the cohesive shared-utility modules (URI templates, tool-name validation, `_meta` metadata, shared auth structs, stdio framing). These are the last L0 pieces; completing S2 unblocks Stage S3 (transport port + in-memory adapter), which imports only L0.

S2's modules split cleanly into testable sub-multi-day units: the validator port (M3) and its default provider (M3) are separated because the provider's documented-keyword-subset work is substantial; the schema-normalization util (M4) is its own unit because it bridges contracts and JSON Schema and delegates to M3; the small M5 utilities (M5a–M5e) are grouped by cohesion and size — URI templates (M5a) and tool-name validation (M5b) each earn their own item against TS test fixtures, while the smaller `_meta` metadata (M5c) and shared-auth (M5d) modules are grouped, and stdio framing (M5e) is its own item with a standalone harness.

**M5e is orphaned until S6a.** `shared/stdio.rkt` (M5e — newline-delimited JSON framing) has **no consumer inside S2**: its first real consumer is the stdio transport (M7) in **S6a**. It is built here for L0 cohesion (architecture groups it under M5) and is **unit-tested standalone** — multi-message and partial-frame buffering against a synthetic byte stream — with integration coverage arriving alongside M7 in S6a. (An implementer who prefers to defer M5e may move it beside M7 in S6a without affecting any other S2 deliverable; this queue builds it now.)

**Prerequisite for the Worker:** the TS reference checkout at `typescript-sdk/` must be available, since several acceptance criteria assert behavioural parity — URI-template round-trips vs `uriTemplate.test.ts`, tool-name accept/reject parity vs the `toolNameValidation` tests, and validator results cross-checked against a TS Ajv-validated baseline. Every S2 module imports **only** S1 (types M1 + errors M2); none may pull in a transport, engine, or role module.

### Item 010: Validator-provider port (M3)
Implement `mcp/core/validators/provider.rkt` — the pluggable JSON-Schema validator-provider interface via `racket/generic` (`gen:`-style), mirroring TS `validators/types.ts`. Define the port operations: **compile** a JSON Schema → a reusable validator handle, and **validate** a value against a compiled handle → an ok / errors result (with a structured error shape carrying path + message). This is the seam that lets the default Racket-native provider (item 011) and any future vetted-library provider be swapped without changing callers. Provide an explicit `provide`d public surface; the module imports only S1 (types + errors). Testable: a unit test under `mcp/core/validators/test/` defines a trivial stub provider satisfying the generic interface, compiles a schema through it, validates an ok and an error value, and asserts the result shape (ok vs structured errors); `raco test` passes.

### Item 011: Default Racket-native provider — `from-json-schema` (M3)
Implement `mcp/core/validators/from-json-schema.rkt` — the default provider implementing the item-010 port over a **documented JSON-Schema subset**, mirroring TS `fromJsonSchema.ts`. Record the hand-rolled-subset-vs-library decision in the module docs as a justified Minimal-deps choice (default = hand-rolled subset unless a vetted lib is adopted). **Minimum supported keyword set:** `type` (`string`/`number`/`integer`/`boolean`/`object`/`array`/`null`), `properties`, `required`, `enum`, `items`, and `format` for the common string formats (`date-time`, `uri`, `email`). Any keyword outside this set is **documented as unsupported** and either ignored-with-warning or rejected explicitly — never silently mis-validated. Testable: a suite under `mcp/core/validators/test/` includes **at least one accept + one reject case for each** of `type`, `object`/`properties`, `required`, `enum`, and `string`-`format`, each cross-checked against a TS Ajv-validated baseline for the same schema + value; a test asserts that an unsupported keyword is handled per the documented policy (and is listed in the module docs); `raco test` passes.

### Item 012: Schema-normalization util — contract-or-JSON-Schema (M4)
Implement `mcp/core/util/schema.rkt` — normalize a `racket/contract` flat contract **or** a JSON Schema into (a) a **wire JSON Schema** for advertisement and (b) a **validation handle** that delegates to the M3 provider (item 010/011). This is the Standard-Schema analogue, mirroring TS `util/schema.ts` + `standardSchema.ts`. Accept both input forms and produce a uniform normalized result the high-level server (S6b) will later consume for tool I/O. Imports S1 + M3 only. Testable: under `mcp/core/util/test/`, a **dual-form** test — a contract input and an *equivalent* JSON-Schema input both produce a validation handle that **accepts the same values and rejects the same values**; a separate assertion confirms the emitted wire JSON Schema matches expectation for each input form; `raco test` passes.

### Item 013: URI templates — RFC 6570 subset (M5a)
Implement `mcp/core/shared/uri-template.rkt` — the RFC 6570 subset used by MCP resource templates, mirroring TS `uriTemplate.ts`: `expand(template, vars) → uri` and `match(template, uri) → vars`. Cover the operators/forms exercised by the TS implementation (simple `{var}`, and the reserved/path-style forms the TS `uriTemplate.test.ts` fixtures use). Imports only S1. Testable: under `mcp/core/shared/test/`, an **expand/match round-trip** test that ports the `typescript-sdk` `uriTemplate.test.ts` fixtures — expand a template with vars and assert the URI, then match that URI back to the original vars — for each fixture form, asserting parity with the TS results (G1); `raco test` passes.

### Item 014: Tool-name validation (M5b)
Implement `mcp/core/shared/tool-name-validation.rkt` — the tool-name predicate + normalizer per spec, mirroring TS `toolNameValidation.ts`. Expose a `valid-tool-name?`-style predicate and the normalizer, applying the same character/length/format rules the TS validator enforces. Imports only S1. Testable: under `mcp/core/shared/test/`, an accept/reject table that asserts the module **accepts and rejects the same names** as the TS `toolNameValidation` tests (port the TS fixture set), including the boundary cases (empty, over-length, illegal characters, valid edge names); `raco test` passes.

### Item 015: `_meta` metadata utils + shared `AuthInfo` (M5c + M5d)
Implement two cohesive shared modules. (1) `mcp/core/shared/metadata-utils.rkt` (M5c) — read/write the **reserved `_meta` keys** (protocol version, client info, client capabilities, related-task, deprecated log level), mirroring TS `metadataUtils.ts`; provide accessors/setters that respect the reserved-key namespace and round-trip with the S1 `_meta` envelope from the `2026-07-28` spec types (item 004). (2) `mcp/core/shared/auth.rkt` (M5d) — the **`AuthInfo` struct** + token/metadata helpers, mirroring TS `auth.ts` + `authUtils.ts`; this is the shared struct the S8 client + server auth will both consume (no OAuth logic here — struct + helpers only). Both import only S1. Testable: under `mcp/core/shared/test/`, (a) a `_meta` test that writes each reserved key, reads it back, and asserts a non-reserved key is left untouched (round-trips against the item-004 `_meta` envelope); (b) an `AuthInfo` test that constructs the struct, exercises the token/metadata helpers, and asserts the field surface matches the TS `AuthInfo` shape; `raco test` passes.

### Item 016: stdio framing — newline-delimited JSON (M5e, orphaned until S6a)
Implement `mcp/core/shared/stdio.rkt` (M5e) — newline-delimited JSON **frame encode/decode over a byte stream**, mirroring TS `stdio.ts`. This is the only M5 module performing I/O. Provide an encoder (message → framed bytes) and a decoder/buffer that consumes a byte stream and yields complete messages, **buffering partial frames** across reads. **Orphaned-until-S6a note:** this module has **no consumer in S2** — its first real consumer is the stdio transport (M7) in S6a; it is built here for L0 cohesion and **unit-tested standalone**, with integration coverage arriving with M7. Imports only S1. Testable: under `mcp/core/shared/test/`, a standalone harness that round-trips a **multi-message** byte stream (encode N messages, feed the concatenated bytes to the decoder, assert all N decode in order) **including partial-frame buffering** (feed the stream split mid-frame across two reads and assert correct reassembly); `raco test` passes.

## Why this batch

Stage S2 (✅) completed layer L0 — validators, schema-normalization, and the shared utils. **Stage S3 opens layer L1** by defining the single hexagonal **transport port** (M6) every concrete adapter will implement, and delivering the **first concrete adapter** — the in-memory paired transport (M10). The in-memory adapter unblocks all downstream engine/role development and testing (S4 protocol engine, S5 roles) without any OS I/O: tests wire two endpoints together in-process. S3 imports only L0 (S1 types/errors + S2 utils) — no subprocess, no socket, no `web-server`.

The stage splits into testable sub-multi-day units: the **port interface** (M6) is its own item because the `gen:transport` contract + callback-sink shape must be settled before any adapter is written; the **in-memory adapter** (M10) is its own item because asynchronous cross-thread delivery is the substantive engineering; the **barrel**, **test suite**, **portability sweep**, and **demo + closeout** each earn separate items matching the S2 house style.

**`related-request-id` is defined-but-inert in S3 (flag, do not "remove").** The M6 port carries a `related-request-id` option on `send`. It is **first load-bearing in S6a/M8** (Streamable HTTP uses it to route a server-initiated request + response onto the correct client-opened SSE stream; essential in S7b for server-initiated sampling/elicitation over HTTP). In-memory (S3) and stdio (S6a) **ignore it** — single bidirectional channel. The port shape is only *validated by a real consumer* once M8 lands. Flagged so the Worker does not mistake the option for dead weight and strip it.

**In-memory delivery MUST be asynchronous (architecture M10 note).** The peer's `on-message` is invoked on a **separate thread**, NOT inline with `send`. A `send` must return **before** the peer handler runs, so ordering/concurrency match a real transport (no synchronous re-entrancy, no head-of-line blocking). This is an explicit acceptance criterion, not an implementation detail.

**Prerequisite for the Worker:** the TS reference checkout at `typescript-sdk/` must be available — `shared/transport.ts` is authoritative for the port method/callback surface and `util/inMemory.ts` for the linked-pair relay semantics. Every S3 module imports **only** L0 (S1 types M1 + errors M2, and S2 utils as needed); none may pull in a subprocess, socket, or `web-server` module.

### Item 019: Transport port — `gen:transport` interface (M6)
Implement `mcp/transport/transport.rkt` — the hexagonal transport port every adapter implements, via `racket/generic` (`gen:transport`), mirroring TS `shared/transport.ts`. **Method surface:** `start` (begin processing / open the channel), `send` (transmit one JSON-RPC message; **options:** `related-request-id`, resumption-token), `close` (tear down). **Callback sinks** (settable handlers, fired by the adapter): `on-message` — receives a message **plus a message-extra-info** value carrying session, auth (`AuthInfo` from S2/M5d), and HTTP-request info; `on-close`; `on-error`. **Optional `session-id`** accessor on the port. Define the `message-extra-info` struct (or contract) here as part of the port surface. **`related-request-id` is defined here but inert until S6a/M8** — document it on the port as a routing hint that in-memory and stdio ignore; do NOT treat as dead weight (see batch note). Provide an explicit `provide`d public surface; imports only L0 (S1 types + errors; S2 `auth.rkt` for `AuthInfo`). Testable: under `mcp/transport/test/`, a unit test defines a trivial stub struct satisfying `gen:transport`, asserts the method surface (`start`/`send`/`close`) is callable, sets each callback sink and confirms it is invokable, and asserts the `message-extra-info` field surface (session, auth, http-req-info) and `session-id` accessor exist; `raco test` passes.

### Item 020: In-memory paired transport — async linked endpoints (M10)
Implement `mcp/transport/in-memory.rkt` — the first concrete adapter implementing the item-019 port, mirroring TS `util/inMemory.ts`. A constructor returns a **linked pair** of endpoints; a message `send` on one endpoint is relayed to the **peer's `on-message`**. **Delivery is asynchronous (load-bearing):** the peer handler runs on a **separate thread** (relay via a `racket/async-channel` or thread + channel), so `send` returns **before** the peer's `on-message` fires — never inline, never re-entrant. The relay must preserve **FIFO ordering per direction** and support **N concurrent in-flight messages with no loss and no head-of-line blocking**. Implement `close` to fire **`on-close` on both endpoints** and tear down the relay; surface relay/encode failures via **`on-error`**. **`related-request-id` option is accepted and ignored** (single bidirectional channel — inert until S6a; document inline). Imports the M6 port (item 019) + L0 only. Testable: deferred to the dedicated suite in item 021 (this item delivers the adapter; item 021 delivers the behavioural tests); a smoke check that the module loads and a pair constructs cleanly via `raco test` is sufficient here.

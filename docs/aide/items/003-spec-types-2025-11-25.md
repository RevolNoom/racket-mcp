## Description

Implement `mcp/core/types/spec-2025-11-25.rkt`, a **pure-data** Racket module mirroring the
`2025-11-25` MCP revision as defined in
`typescript-sdk/packages/core/src/types/spec.types.2025-11-25.ts`. It provides a Racket
**struct** + a **flat contract** (`racket/contract`) for **every** request, notification,
result/response, error, and the supporting payload types of that revision, plus
`read-json → struct → write-json` (de)serialization that round-trips a TS-SDK-shaped JSON
message back to byte-equivalent JSON.

This module sits one level above `constants.rkt` (item 001) and `guards.rkt` (item 002) in
the M1 type layer. The guards classify a *generic* JSON-RPC envelope; this module adds the
**MCP-method-specific** payloads (`initialize`, `tools/call`, `resources/read`, …) as typed
structs. It is consumed downstream by **item 004** (the sibling `2026-07-28` revision —
designed to be structurally parallel so the two can be diffed) and **item 005** (the **N1
normalized-superset façade** in `types.rkt`, which will UNION 003 + 004 into one
version-agnostic shape). **Design constraint (from N1, architecture line 326):** 003 must
expose a clean, enumerable, per-primitive surface so 005 can union it field-by-field with
004's parallel primitives — see *Decisions & Trade-offs → N1-readiness*.

This is a **large** module (≈ 70 named types). The spec below **enumerates every one** in a
concrete inventory table (§Type inventory). The implementer MUST NOT "mirror the file" by
eyeballing it — each row is the contract. The types are grouped by category
(JSON-RPC envelope, common, lifecycle/initialize, ping, progress/cancellation,
pagination, resources, prompts, tools, tasks, logging, sampling, autocomplete, roots,
elicitation, content blocks, model preferences) to keep the module navigable, but **all are
required**.

### Representation conventions (inherited from item 002 / `read-json`)

These are the **wire** conventions every (de)serializer in this module must obey; they are
already established by `guards.rkt` and `constants.rkt` and are non-negotiable for parity:

- A JSON **object** is the `read-json` shape: an **immutable `hasheq` with symbol keys**
  (`json-object?` from guards — re-implement the same predicate internally; do not depend on
  guards' un-provided helper).
- JSON **null** is the symbol `'null`. JSON arrays are Racket lists. JSON strings/numbers/
  booleans are Racket strings / numbers / `#t`/`#f`.
- An **absent optional field** is NOT `'null`. The struct field carries a Racket sentinel
  (recommend the symbol `'absent`, exported as a binding `MISSING`/`absent` so 005 can use
  the same sentinel) — and the serializer **omits the key entirely** (it must NOT emit
  `"key": null`). Conversely a present `'null` value (rare here; see `Task.ttl` which is
  `number | null`) is serialized as JSON `null`. This absent-vs-null distinction is the
  single most parity-sensitive serialization rule and is called out per-field in the
  inventory where `| null` appears.
- `JSONRPC-VERSION` comes from `constants.rkt`; the error `code` constants
  (`PARSE-ERROR` … `URL-ELICITATION-REQUIRED`) come from `constants.rkt`.
- **Strictness — THREE distinct behaviors (parity-corrected; verify against schemas.ts):**
  1. The JSON-RPC *envelope* schemas are `.strict()` (no extra top-level keys —
     schemas.ts:147,157,168,192), exactly as item 002 enforces. Extra top-level key →
     **rejected**.
  2. **Results are genuinely LOOSE.** `ResultSchema = z.looseObject` (schemas.ts:118) and
     every result derives from it (`.extend`/`.merge`/`.loose()`), so a result preserves
     `_meta` AND arbitrary unknown extra keys verbatim. `GetTaskPayloadResult` is fully open
     (`ResultSchema.loose()`, schemas.ts:753). Capability sub-trees are `looseObject`
     (schemas.ts:351+).
  3. **Concrete request/notification PARAMS are NOT loose — they STRIP unknown non-`_meta`
     keys.** This corrects an earlier draft that marked all params "(loose)". The base is
     `BaseRequestParamsSchema = z.object({...})` — a plain (non-loose) object,
     schemas.ts:78. Each concrete params type is `BaseRequestParamsSchema.extend({...})`
     (e.g. `InitializeRequestParamsSchema`, schemas.ts:463), which **drops** unknown keys
     that are neither a named field nor `_meta`. **Only the GENERIC envelope's untyped
     `params` is `BaseRequestParamsSchema.loose()`** (schemas.ts:102) — that is the
     pass-through used by the raw `jsonrpc-request` envelope, NOT by the typed concrete
     params structs. So: a typed `initialize-request-params` with a stray unknown key
     **drops** that key on round-trip (matching TS); a `result` with a stray key
     **preserves** it. `_meta` is a NAMED field on both, so its passthrough is fine either
     way.

### `_meta` and additionalProperties passthrough (parity-critical)

Results (and the capability trees and `GetTaskPayloadResult`) permit a `_meta` object AND
arbitrary extra keys (`ResultSchema = z.looseObject`, `GetTaskPayloadResult` fully open,
schemas.ts:118,753). A struct-per-type mapping that only models the *named* fields would
**silently drop** `_meta` and unknown keys on round-trip, breaking parity. **Decision (bake
into the spec):**
- Every struct whose TS type derives from `Result` (every result type), is a `looseObject`,
  or is a capability tree carries a dedicated **`rest`** field (an immutable `hasheq` of the
  leftover keys, including `_meta`). Deserialize splits known keys → struct fields and
  unknown keys → `rest`; serialize merges `rest` back in. The inventory marks each such type
  **(loose: has `rest`)**.
- **Request/notification params structs do NOT carry a `rest` for unknown non-`_meta`
  keys** — they have a named `meta` field (for `_meta`) but **drop** other unknown keys on
  deserialize and do NOT re-emit them, matching TS's non-loose `BaseRequestParamsSchema`. The
  inventory marks these **(params: drops unknown; `meta` named only)**. (`_meta` itself is
  preserved via the named `meta` field; `task`/`progressToken`/`io.modelcontextprotocol/related-task`
  inside `_meta` are part of `RequestMetaSchema`, schemas.ts:64 — keep them inside `meta`.)

## Acceptance criteria

- [ ] `mcp/core/types/spec-2025-11-25.rkt` exists as `#lang racket/base` with
      `(require racket/contract)` and an explicit curated `(provide …)` (no
      `(all-defined-out)`; no `(contract-out)`-free raw exports of the constructors that need
      contracts — see Decisions on whether to `contract-out` or provide raw + `…/c`).
- [ ] **Every** type in §Type inventory (A–Q) is implemented: a transparent `struct` per
      object type, a flat contract (`…/c`) per type (including the string-enum and union
      contracts), with kebab-case field names mapped to the exact JSON camelCase keys.
- [ ] Structs are `#:transparent` (equality + readable printing for tests). Predicates
      (`initialize-request?` etc.) are provided.
- [ ] A per-type **flat contract** exists and is enforceable (`contract?` true); each
      contract rejects malformed inputs (wrong type, missing required field, wrong literal
      `type`/`method`, out-of-enum value).
- [ ] **Deserialization** `(json->X jsexpr)` for each top-level message type maps a
      `read-json` jsexpr (symbol-keyed `hasheq`, `'null`, lists) into the struct, splitting
      unknown/`_meta` keys into the `rest` field for loose types. Raises (or returns a
      contract violation) on malformed input.
- [ ] **Serialization** `(X->json struct)` produces a `write-json`-ready jsexpr that:
      - omits absent optional fields entirely (NO `"k": null` for absent),
      - emits a present `'null` (only `Task.ttl`) as JSON null,
      - merges the `rest`/`_meta` keys back verbatim,
      - maps kebab-case fields back to the exact camelCase / `$schema` / `_meta` JSON keys.
- [ ] **Round-trip parity:** for a representative message of **each envelope kind** (request,
      notification, result-response, error-response) — at minimum `initialize` request,
      `tools/call` request, `initialize` result, a `notifications/progress` notification, a
      `tools/list` result with pagination + `_meta`, and an error response — parsing the
      fixture JSON to a struct and re-serializing yields JSON that is **semantically
      identical** to the fixture (see Testing Strategy for the canonicalized comparison —
      compare re-parsed jsexpr, not raw bytes).
- [ ] **`_meta`/extra-key passthrough (RESULTS ONLY):** a **result** fixture carrying `_meta`
      and an unknown extra key round-trips with BOTH preserved (the key lands in `rest` and is
      re-emitted). **Request/notification PARAMS do the opposite:** a params fixture carrying
      an unknown non-`_meta` key round-trips with that key **DROPPED** (matching TS's non-loose
      `BaseRequestParamsSchema`); only its `_meta` (named `meta` field) survives.
- [ ] **Contract-rejection:** for each major category, a malformed input (e.g. `initialize`
      missing `protocol-version`; `tools/call` with non-string `name`; `set-level` with an
      out-of-enum `level`; a content block with `type:"text"` but missing `text`; `Task`
      missing required `ttl`) is rejected by its contract / deserializer.
- [ ] **Three-way strictness parity:** (a) an envelope with an extra top-level key is
      **rejected** (strict envelope); (b) a **result** with an extra inner key is **accepted
      and preserved** (loose result); (c) a concrete **request/notification params** with an
      extra non-`_meta` key is **accepted but the key is DROPPED** on re-serialize (non-loose
      `BaseRequestParamsSchema`, schemas.ts:78). Do NOT assert preservation on params.
- [ ] The four error codes / `JSONRPC-VERSION` are imported from `constants.rkt` (not
      re-literaled); `URLElicitationRequiredError` uses `URL-ELICITATION-REQUIRED`.
- [ ] `raco test mcp/core/types/` passes (exit 0) from repo root — module + test compile and
      load cleanly alongside items 001/002.
- [ ] **Portability (NFR):** the module requires only `racket/base`, `racket/contract`,
      `json` (for the jsexpr conventions; no I/O at module load), and `constants.rkt`. No
      subprocess/socket module. (Restricted-load test is item 008's job; this module must not
      introduce such a dependency.)
- [ ] **N1-readiness:** the per-primitive structs and union contracts are individually
      provided (not hidden behind a single opaque blob) so item 005 can union 003 + 004

      under `core/types/*` is advanced toward `partial` on completion (Completion Reminder);

## Testing strategy

**Test file:** `mcp/core/types/test/spec-2025-11-25-test.rkt` (`#lang racket/base`,
`require rackunit`, `json`, `racket/runtime-path` for fixture paths, the module under test).
The test has four parts.

### Fixture source decision (CALLED OUT)

There are **no ready-made JSON fixtures** in the TS checkout for round-tripping:
- `specTypeSchema.examples.ts` is **JSDoc snippet bait** (40 lines; just
  `specTypeSchemas.CallToolResult[...].validate(untrusted)` illustrations) — **NOT** JSON
  data. Do not use it as a fixture source.
- `test/spec.types.2025-11-25.test.ts` is a **static type-assignment** test
  (`sdk = spec; spec = sdk` mutual-assignability checks, 950 lines) — **not** JSON fixtures.
- No `*.json` fixture files exist under `packages/core` for these types.

**Therefore the implementer authors the fixtures by hand**, modeling them on the exact TS
type shapes (each field a valid example value), and stores them as `.json` files under
`mcp/core/types/test/fixtures/`. To keep them honest, the test also (a) re-parses both sides
and (b) validates each fixture against its struct's contract, so a fixture that drifts from
the type shape fails. (Optional, recommended cross-check: where the TS test or
`specTypeSchema.ts` registry can be invoked, confirm field names match — but that requires a
node toolchain and is out of scope for `raco test`; note it as a manual step.)

### Part 1 — round-trip per envelope kind (the queue's core requirement)

For each fixture file `F` (at minimum: `initialize-request.json`, `tools-call-request.json`,
`initialize-result.json`, `progress-notification.json`, `list-tools-result.json` (with
`_meta`, `nextCursor`, and a tool with `inputSchema`), `error-response.json`, **plus one
intersection-type fixture `get-task-result.json`** (`Result & Task` — all `Task` fields at the
top level + `_meta`, proving the flatten round-trips losslessly — see §J TRAP), **plus one
discriminated-union fixture per union family** (see the union-coverage note below)):

1. `(define orig (call-with-input-file F read-json))`.
2. `(define s (json->X orig))` — parse to struct.
3. `(define rt (X->json s))` — re-serialize.
4. **Canonicalized equality:** assert `(jsexpr=? orig rt)` where `jsexpr=?` is a recursive
   comparator that treats JSON objects as **unordered key sets** (compare `hasheq`s by key,
   not by `write-json` byte order — JSON object key order is not guaranteed and `write-json`
   ordering need not match the fixture's). Lists compare in order; `'null` compares to
   `'null`. **Do NOT compare raw bytes** — that would falsely fail on key reordering.
   (Document this in Decisions: "identical" = jsexpr-equal under unordered object keys.)
5. As a secondary belt: assert `(X->json (json->X rt))` equals `rt` (idempotent second pass).

**Discriminated-union coverage — HARD MINIMUM: ONE round-trip fixture (or fixture sub-value)
PER ARM.** The deserializer dispatches on a discriminator key; the test MUST exercise EACH
arm so no arm is silently mis-dispatched or dropped. The minimum arm count is **23** distinct
arm fixtures across these families (a generic "ContentBlock" test does NOT cover the
sampling-only `tool_use`/`tool_result` arms — those need their own fixtures):
- **`ContentBlock`** (line 1732, discriminator `type`) — **5 fixtures:** `text`, `image`,
  `audio`, `resource_link`, `resource` (embedded). E.g. a `prompt-message` /
  `call-tool-result` `content` list with one of each.
- **`SamplingMessageContentBlock`** (line 1690) — **5 fixtures:** `text`, `image`, `audio`,
  **`tool_use`**, **`tool_result`** — the last two NEVER appear in the general ContentBlock
  test, so they get dedicated fixtures here. ALSO round-trip a `sampling-message` whose
  `content` is a single block AND one whose `content` is a LIST (TS `Block | Block[]`, line
  1680) — 2 extra shape fixtures.
- **`PrimitiveSchemaDefinition`** (line 2224, discriminator `type`) — **4 fixtures:**
  `string` (`string-schema`), `number`, `integer` (both via `number-schema`), `boolean`.
- **`EnumSchema`** (line 2441) — **5 fixtures:** untitled-single (`enum`), titled-single
  (`oneOf`), untitled-multi (`type:"array"`+`items.enum`), titled-multi (`items.anyOf`),
  legacy-titled (`enumNames`). Carry each inside an `elicit-request-form-params`
  `requestedSchema.properties`.
- **`ElicitRequestParams`** (line 2206, discriminator `mode`) — **2 fixtures:** `form`, `url`.
- **`ResourceContents`** (discriminator: `text` key vs `blob` key) — **2 fixtures:**
  `TextResourceContents`, `BlobResourceContents` (one of each in a `read-resource-result`
  `contents` list).

Each arm fixture round-trips via the Part-1 `jsexpr=?` procedure AND asserts it deserialized
to the EXACTLY-correct struct predicate (e.g. `(tool-use-content? …)`), so a mis-dispatch
fails loudly.

**Populated nested-capability fixture (HARD MINIMUM — not just shallow `roots.listChanged`).**
Add a fixture exercising the DEEP capability tree: an `initialize-request` whose
`ClientCapabilities` populates `tasks.requests.sampling.createMessage` and
`tasks.requests.elicitation.create` (lines 363–382) AND `sampling.{context,tools}` AND
`elicitation.{form,url}` AND `experimental`; and an `initialize-result` whose
`ServerCapabilities` populates `tasks.requests.tools.call` (lines 450–460), `prompts`,
`resources.{subscribe,listChanged}`, `tools`, `logging`, `completions`. Assert the deep
nested values survive round-trip (they live in the loose capability `hasheq` per the Decisions
verdict — see #Decisions "deep capability trees = loose hasheq").

### Part 2 — `_meta` / additionalProperties passthrough (scoped: RESULTS preserve, PARAMS drop)

- **Results preserve.** Use a **result** fixture (e.g. `list-tools-result.json`) carrying
  `_meta:{progressToken:…}` plus an unknown extra top-level key in the loose result. Assert
  after round-trip the re-serialized jsexpr still contains `'_meta` AND the unknown key (they
  landed in `rest` and were merged back). Also assert a content block carrying `_meta`
  preserves it.
- **Params DROP (the parity correction).** Use a **request-params** fixture (e.g.
  `initialize-request.json`) carrying `_meta` AND an unknown non-`_meta` top-level key in
  `params`. Assert after round-trip the `_meta` SURVIVES (named `meta` field) but the unknown
  key is **GONE** (concrete params are non-loose `BaseRequestParamsSchema.extend`,
  schemas.ts:78,463 — they strip unknowns, matching TS). This is the deliberate
  asymmetry; do NOT assert the unknown key survives on params.

### Part 3 — contract-rejection (the queue's second requirement)

Enumerate at least one reject case per category, each its own named `check-exn`/contract
check:
- `initialize` params missing `protocolVersion` → rejected.
- `tools/call` params with `name` = number → rejected.
- `set-level` with `level:"verbose"` (out of `logging-level/c`) → rejected.
- content block `{type:"text"}` missing `text` → rejected; `{type:"bogus"}` → union dispatch
  fails / rejected.
- `Task` missing required `ttl` → rejected; `Task` with `status:"running"` (out of enum) →
  rejected; `Task` with `ttl:'null` → **accepted** (nullable).
- `jsonrpc-request` with an extra top-level key → rejected (strict envelope).
- `image-content` missing required `mimeType` → rejected.
- an `id` of `'null` / `1.5` where a request-id is required → rejected (reuse guards' rule).

### Part 4 — three-way strictness parity (envelope / result / params)

- **Envelope** with extra top-level key → **rejected**.
- **Result** with an extra inner key → **accepted AND preserved** (Part 2 asserts
  preservation; here assert acceptance — a loose result must not reject).
- **Concrete request/notification params** with an extra non-`_meta` key → **accepted but the
  key DROPPED** on re-serialize (non-loose `BaseRequestParamsSchema`, schemas.ts:78). Assert
  the round-tripped params jsexpr does NOT contain the unknown key. (Do NOT use a result-style
  "preserved" assertion here — this is the deliberate result-vs-params asymmetry.)
- `URLElicitationRequiredError` round-trips with `error.code = -32042` (assert the value
  equals `URL-ELICITATION-REQUIRED` from `constants.rkt`) and `error.data.elicitations` a
  list of URL-elicit params.

### Part 5 — field-name mapping unit test (anti-vacuous; INDEPENDENT of fixtures)

A standalone test (no fixture files) that pins the kebab↔camel and literal-key mappings
directly, so a fixture that happens to round-trip by luck cannot mask a wrong field map:
- Construct a struct in Racket with known field values; serialize; assert the output `hasheq`
  has the EXACT JSON keys: `serverInfo` (not `server-info`/`serverinfo`), `inputSchema`,
  `protocolVersion`, `clientInfo`, `nextCursor`, `mimeType`, `uriTemplate`, `isError`,
  `structuredContent`, `toolUseId`, `_meta` (the named `meta` field serializes to literal
  `_meta`), and `$schema` (preserved verbatim — see verdict: `$schema` stays inside `rest`).
- Deserialize a hand-built jsexpr with those exact camelCase keys; assert each maps to the
  right struct field.

## Decisions & Trade-offs

**Implementation decisions recorded 2026-06-17 (item 003 delivered):**
- **`contract-out` vs raw struct + `…/c`:** CHOSE raw transparent structs + separate
  provided `…/c` flat contracts (no `contract-out`). Item 005's façade constructs/destructures
  freely; per-call enforcement belongs at the protocol boundary. `(struct-out …)` exports the
  constructor/predicate/accessors; each `name/c` is provided alongside.
- **Envelope struct `params` field renamed to `payload`/`body` accessor where it collided:**
  a struct `X-request` with a `params` field auto-generates accessor `X-request-params`, which
  collides with the sibling `X-request-params` struct type. The wire key stays `params`; only
  the Racket accessor differs (`X-request-payload`). Affects the request/notification envelope
  structs that have a matching `…-params` struct. `tool`'s `execution`/`annotations` fields use
  accessors `tool-exec`/`tool-annots` to avoid colliding with the `tool-execution`/
  `tool-annotations` struct types. Field-mapping is unaffected (Part-5 unit test pins it).
- **`lit/c` string-literal contract:** `(=/c "…")` is NUMERIC equality and rejects strings;
  added a `lit/c` helper (`equal?`-based flat contract) for all `method`/`type`/`mode`
  discriminator pins.
- **Capabilities modeled as a single `rest` hasheq:** `client-capabilities`/`server-capabilities`
  are top-record structs wrapping the WHOLE capability object as one loose `hasheq` (`rest`
  field). This is the cleanest lossless realization of "deep trees = loose blobs, structs only
  for the top record" — deep `tasks.requests.sampling.createMessage` values round-trip verbatim
  (proven by the deep-capability fixture).
- **Definition ordering:** content-block union + structs are defined BEFORE prompts/sampling
  (they reference `content-block/c` at load time); the `content-block/c` union is defined before
  `tool-result-content/c` (which references it).
- **Intersection flatten:** `get-task-result`/`cancel-task-result` (`Result & Task`) and
  `task-status-notification-params` (`NotificationParams & Task`) carry the 7 Task fields at the
  struct top level + `meta` + `rest`; `read-task-fields`/`emit-task-fields`/`read-task-rest`
  centralize the flatten. Proven lossless by `get-task-result.json`.
- **N1-readiness field-presence model:** an optional field absent on the wire holds the exported
  `absent` sentinel; serialize omits it. A present `'null` (only `Task.ttl`) is emitted as JSON
  null. Item 005 unions by testing `(absent? v)`; the `absent`/`absent?`/`present?` bindings are
  provided for that.

**Original open decisions (now settled — preserved for the record):**

- **`contract-out` vs raw struct + `…/c`:** whether to `(provide (contract-out [struct …]))`
  (contracts enforced at the boundary, slower) or provide raw transparent structs plus
  separate `…/c` flat contracts the caller applies (faster, more flexible for 005's façade).
  *Recommendation:* provide raw transparent structs + separate `…/c` contracts (item 005
  needs to construct/destructure freely; per-call contract checking belongs at the protocol
  boundary, not here). Record the final choice.
- **Loose-type `rest` field:** confirmed approach for `_meta`/additionalProperties
  passthrough (every **result** / `looseObject` / capability tree gets a `rest` `hasheq`).
  Alternative (rejected): enumerate `_meta` contents — impossible, it's open.
- **Result vs params strictness (SETTLED — parity correction):** **results are loose** (carry
  `rest`, preserve unknown keys); **concrete request/notification params are NOT loose** —
  `BaseRequestParamsSchema = z.object` (schemas.ts:78), and each concrete params type is
  `.extend` of it (schemas.ts:463), so unknown non-`_meta` keys are **dropped** on round-trip.
  Params structs carry a named `meta` field (for `_meta`) but NO `rest`. Only the generic
  envelope's untyped `params` is `.loose()` (schemas.ts:102) — that pass-through belongs to the
  raw `jsonrpc-request` envelope, not the typed params structs. The round-trip/passthrough
  tests are scoped accordingly (results preserve; params drop).
- **Canonical equality for round-trip:** `jsexpr=?` compares objects as unordered key sets,
  lists in order, numbers by `=`, `'null` by `eq?`. Raw-byte comparison rejected (JSON key
  order not guaranteed; `write-json` ordering need not match the fixture).
- **`$schema` / `_meta` key mapping (SETTLED):** `_meta` maps to a named `meta` field
  (serializes back to literal `_meta`). `$schema` is **kept inside the open JSON-Schema
  fragment** — model `Tool.inputSchema`/`outputSchema` and `requestedSchema` as a loose
  `hasheq` (NOT a struct field), so `$schema` (and all other JSON-Schema keywords) survive
  verbatim. A standalone field-mapping unit test (Testing Part 5) pins serverInfo/inputSchema/
  uriTemplate/etc. independently of fixtures.
- **Nested capability sub-objects (SETTLED):** deeply-nested capability trees
  (`ClientCapabilities.tasks.requests.sampling.…`, schemas.ts:351+) are modeled as nested
  loose `hasheq` blobs; explicit structs only for the top `client-capabilities` /
  `server-capabilities` record. A POPULATED nested-capability fixture (Testing Part 1) proves
  the deep values round-trip.
- **Interface-intersection results (`GetTaskResult`/`CancelTaskResult = Result & Task`,
  `TaskStatusNotificationParams = NotificationParams & Task`):** flatten the Task fields into
  the result/params struct (NOT a `task` sub-struct) — Task fields appear at the wire top
  level; see the §J TRAP callout. A `get-task-result.json` round-trip fixture proves the
  flatten is lossless.
- **`EmptyResult` strictness (SETTLED):** schemas.ts:207 makes it strict, but for S1 it is
  treated as an ordinary `result` (no strict enforcement, no stray-key fixture on it). This is
  a **known, deliberately-deferred passthrough gap** — full `.strict()` enforcement can be
  added later additively.
- **N1-readiness contract:** document the field-presence model (absent sentinel) that item
  005's façade unions against, so 004 mirrors it exactly.
